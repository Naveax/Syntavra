# SignalCore

SignalCore is a local-first runtime-control plane and Agent Skill. Correctness, exact evidence, security boundaries and required verification outrank token reduction.

## Activation contract

1. Negotiate the real host capabilities. Never describe instruction-only operation as enforced runtime operation.
2. Initialize project-scoped state, exact evidence, broker, structural index, memory and rollout telemetry.
3. Retrieve exact definitions and transitive impact before broad search or semantic expansion.
4. Route long commands through the durable broker. Model-mediated wait loops are forbidden when runtime control is available.
5. Keep full output in exact content-addressed evidence; inject only bounded summaries with critical markers and recovery handles.
6. Pack context with mandatory task, evidence, impact and verifier roles before optional material.
7. Bind verifier reuse to repository tree, environment, dependencies, toolchain and affected paths.
8. Write immutable history and one canonical summary-DAG root before controlled handoff.
9. Reject 5X or competitor-superiority claims without observed paired quota evidence and a passing receipt.

## Runtime entry points

```bash
python -m signalcore_runtime --project . --host codex status
python -m signalcore_runtime --project . mcp serve
```

Use `python -m signalcore_runtime --help` for the full surface.

## Host compatibility

Native Agent Skill discovery, hooks, MCP and proxy capabilities are independent. SignalCore reports the negotiated mode as `HOOK_ENFORCED`, `MCP_CONTROLLED`, `PROXY_CONTROLLED`, `INSTRUCTION_ONLY` or `UNSUPPORTED`. The maintained compatibility matrix is at [`../../COMPATIBILITY.md`](../../COMPATIBILITY.md).

## Locked domain profiles

The `roblox_studio` profile remains hidden and fail-closed. It requires an authorized bridge, signed short-lived single-use activation and process attestation. Ordinary CLI, IDE and prompt-based profile activation remains rejected.

## Claim boundary

Configured 20X/30X/100X workload shape is not performance proof. A 5X receipt requires observed workload measurements, at least 10 fair paired repetitions, actual quota telemetry, equal verified work, identical execution identity, no verifier skips and no security regression.
